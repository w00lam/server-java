# Kafka 도입 및 이벤트 중심 아키텍처 이해

## 💡 이벤트 중심의 사고방식 (Event-Driven Mindset)

기존 시스템이 **"A가 B에게 일을 시킨다 (Request-Response)"** 는 방식이었다면, 카프카를 도입한다는 것은 **"A가 발생한 일을 알리고, 필요한 사람(B, C, D)이 알아서 가져간다"** 는 방식으로의 전환을 의미합니다. 이는 시스템 전체 관점에서 비즈니스 확장을 매우 유연하게 만듭니다.

---

## ✅ Kafka의 장단점

| 구분 | 내용 |
| :--- | :--- |
| **장점** | **고성능/고가용성**: 초당 수백만 건의 데이터 처리 가능, 서버 장애 시에도 데이터 유실 방지 <br>**확장성**: 브로커 추가를 통해 수평적 확장(Scale-out) 용이 <br>**디커플링**: 생산자와 소비자가 서로 독립적으로 동작 가능 |
| **단점** | **복잡도**: 인프라 설치 및 관리 운영 리소스 증가 <br> **학습 곡선**: 파티션, 오프셋, 컨슈머 그룹 등 전문 개념 숙지 필요 <br> **지연 시간**: 극미세 실시간성(Hard Real-time)이 필수인 경우 오버헤드 발생 가능 |

## 🏗️ 핵심 구성 요소

* **Producer**: 이벤트를 발행하는 주체 (예: 예약 엔진)
* **Consumer**: 이벤트를 구독하고 처리하는 주체 (예: 알림 서비스, 통계 서비스)
* **Topic**: 메시지가 저장되는 논리적인 카테고리 (우편함)
* **Partition**: 토픽을 쪼개서 저장하는 단위 (병렬 처리의 핵심)
* **Broker**: 카프카 서버 그 자체 (보통 3대 이상으로 클러스터 구성)
