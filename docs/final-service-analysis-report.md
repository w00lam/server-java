## 1.시스템 전반 구조 (System Architecture)
우리 서비스는 현대적인 컨테이너 기반 아키텍처를 채택하여 격리성과 확장성을 동시에 확보하고 있습니다.
- **인프라 구성 :** Docker Compose를 활용하여 애플리케이션, Database(MySQL), Cache(Redis)를 독립된 환경으로 운영합니다.
- **요청 흐름 :** 사용자 요청은 API Gateway를 거쳐 Spring Boot 서버로 전달되며, 좌석 예약과 같은 민감한 작업은 DB 트랜잭션을 통해 데이터 정합성을 보장합니다.
- **효율성 :** 리소스 모니터링을 통해 불필요한 자원 낭비를 최소화하고, 트래픽에 따라 유연하게 대응할 수 있는 구조를 갖추고 있습니다.

---

## 2. 서비스 수용 능력 (Service Capacity)
실제 부하 테스트(k6) 데이터를 기반으로 분석한 우리 서비스의 현재 체급입니다.

|주요 지표|측정 결과|비즈니스 의미|
|:---|:---|:---|
|최대 TPS (초당 처리량)|약 37 req/s|초당 37건의 예약 요청을 누락 없이 접수 가능|
|평균 응답 속도 (Latency)|18ms ~ 24ms|사용자 클릭 후 0.03초 이내에 결과 확인 가능 (매우 쾌적)|
|안정적 동시 접속자|50 VUs (가상 사용자)|50명이 동시에 예약 버튼을 눌러도 시스템 마비 없음|
**결론 :** 현재 시스템은 소규모~중규모 공연 예약 시나리오를 안정적으로 소화할 수 있는 성능을 확보하고 있습니다.

## 3. 시스템 취약점 및 개선 과제 (Vulnerabilities)
철저한 분석 결과, 하드웨어 사양보다는 소프트웨어 로직 측면의 병목이 확인되었습니다.
### **🔍 핵심 취약점 :** DB Lock 경합
- **현상 :** 서버 사양(CPU/Memory)을 4배 증설했음에도 처리량(TPS)이 향상되지 않음.
- **원인 :** 한정된 좌석에 대한 동시 수정 요청 시 발생하는 DB 트랜잭션 대기 시간이 성능의 발목을 잡고 있음.
- **개선 방향 :**
    - **단기 :** 현재의 500 에러를 409(Conflict)로 정교하게 예외 처리하여 서버 부하 경감.
    - **장기 :** Redis 분산 락(Distributed Lock) 도입 및 메시지 큐(Kafka)를 활용한 비동기 예약 시스템으로 전환하여 처리량 획기적 개선 예정.

## 4. 장애 대응 매뉴얼 (Incident Response)
어떤 상황에서도 서비스의 "최소 기능"은 유지될 수 있도록 단계별 대응 체계를 구축했습니다.
**✅ 상황별 대응 시나리오**
1. **예측 가능한 부하 (인기 공연 오픈 직후)**
   - **조치 :** API Gateway에서 초당 요청 수 제한(Rate Limiting)을 활성화하여 시스템 전체 다운을 방지합니다.
   - **사용자 안내 :** "대기 순번" 페이지를 노출하여 사용자 이탈을 방지합니다.
2. **예측 불가능한 돌발 장애 (시스템 프리징)**
   - **조치 :** `docker update` 명령을 통한 즉각적인 리소스(CPU/Mem) 긴급 수동 증설.
   - **복구 :** 서비스 불능 시 서킷 브레이커(Circuit Breaker)를 작동시켜 DB 연결을 차단하고, 순차적 재시작을 통해 세션을 복구합니다.

## 5. 최종 제안 및 비전
현재 우리 서비스는 **최소한의 비용(0.5 CPU)으로 최대의 효율**을 내고 있습니다. 무분별한 서버 증설 대신 정확한 병목 지점을 파악했기에, 앞으로는 **기술적 최적화(Redis/Async)** 를 통해 더 적은 비용으로 더 많은 사용자를 수용하는 '강한 시스템'으로 진화할 것입니다.
