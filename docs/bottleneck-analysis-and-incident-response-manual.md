# 시스템 병목 분석 및 장애 대응 매뉴얼
## 1. 테스트 환경 및 스크립트 개요
본 테스트는 **50명의 가상 사용자(VU)** 가 50개의 한정된 좌석을 선점하기 위해 동시에 예약 API(`POST /reservations`)를 호출하는 시나리오로 진행되었습니다.

**주요 테스트 스크립트 설정**
- **도구 :** k6

- **부하 패턴 :** 10초간 50명까지 증가(Ramp-up) → 20초 유지 → 10초 감소

- **검증 로직 :** HTTP 200/201(성공) 및 409(경합 발생) 상태 코드 확인

## 3. 병목 지점 탐색 및 개선 결과 분석
### **🔍 병목 탐색 결과**
- **자원 증설의 무의미성 :** CPU와 메모리를 4배 늘려도 TPS가 약 37/s에서 정체되었습니다. 이는 애플리케이션의 계산 능력 부족이 아닌, **DB 트랜잭션 경합(Locking)** 또는 **커넥션 풀(HikariCP)의 임계치**가 원인임을 시사합니다.
- **응답 처리의 한계 :** 50개의 좌석이라는 한정된 자원에 대해 1,500건의 요청이 몰리면서, 성공한 50건을 제외한 나머지가 모두 `500 Error`로 처리되었습니다. 비즈니스 로직상 `409 Conflict`로 처리되어야 할 부분이 서버 내부 에러로 분출되고 있는 점이 가장 큰 병목이자 개선 포인트입니다.

### **💡 적정 배포 스펙 제안 : [CPU 0.5 / Memory 512M]**
- 테스트 결과 상위 스펙에서도 성능 향상이 없으므로, 현재 로직에서는 **최소 사양**으로 운영하는 것이 비용 효율적입니다. 성능 향상을 위해서는 자원 증설보다 **로직 고도화(Redis 활용 대기열 등)** 가 우선되어야 합니다.

## 4. 장애 대응 매뉴얼 (Incident Response Manual)
### **🚨 [Scenario A] 예측 가능 및 완벽 개선이 어려운 포인트**
**상황 : 특정 좌석에 대한 고도 경합으로 인한 응답 지연 및 에러 발생**
- **원인 :** 현재 DB Lock 메커니즘으로 인해 다수의 사용자가 동일 자원을 수정하려 할 때 대기 시간이 발생합니다.
- **대응 절차 :** 
  - **Rate Limiting 강화 :** 서버 임계치 이상의 요청이 들어오지 않도록 API Gateway 레벨에서 차단합니다.
  - **User-Friendly 에러 처리 :** 현재 발생하는 500 에러를 클라이언트 단에서 "잠시 후 다시 시도해 주세요"라는 메시지로 치환하여 사용자 경험을 보호합니다.
  - **대기열(Queue) 유도 :** 트래픽이 몰리는 특정 ConcertId에 대해 진입 전 대기 순번표를 부여하도록 로직을 전환합니다.
### **⚠️ [Scenario B] 예측하지 못한 돌발 포인트**
**상황 : 자원 증설(Case 2) 중 발생한 Max Latency(1.52s) 스파이크 및 시스템 프리징**
- **원인 :** JVM GC(Garbage Collection) 활동 폭주 또는 컨테이너 리소스 재할당 중의 컨텍스트 스위칭 과부하.
- **대응 절차 :**
  - **즉시 모니터링 :** `docker stats`와 애플리케이션 로그(OOM 여부)를 즉시 확인합니다.
  - **긴급 리소스 업데이트 :** 시스템 프리징 시 `docker update --cpus 2.0` 등 한 단계 높은 사양으로 즉시 명령어를 실행하여 숨통을 틔워줍니다.
  - **서킷 브레이커(Circuit Breaker) 수동 작동 :** DB 부하가 감당 불가능할 경우, 예약 API를 일시적으로 닫아(Fallback) 전체 시스템이 셧다운되는 것을 방지합니다.
  - **트래픽 셰이딩(Traffic Shedding) :** 비로그인 사용자나 우선순위가 낮은 요청부터 순차적으로 거부하여 핵심 비즈니스 기능을 유지합니다.
