# Redis 기반 캐싱 전략 적용 및 성능 개선 보고서

## 1. 과제 개요

본 과제는 애플리케이션의 요청 처리 성능을 향상시키기 위해
**조회가 빈번하고 중복 요청 가능성이 높은 구간을 분석**하고,
해당 구간에 **Redis 기반 캐싱 전략을 적용하여 성능 개선 효과를 검증**하는 것을 목표로 한다.

이를 위해 캐시 적용 전·후의 성능을 비교 측정하고,
그 결과를 바탕으로 캐시 전략의 유효성을 분석하였다.

---

## 2. 테스트 대상 선정 이유

본 성능 테스트에서는 시스템의 모든 기능을 대상으로 하지 않고,
**콘서트 날짜 조회**와 **콘서트 좌석 조회** 두 가지 시나리오로 테스트 범위를 한정하였다.

그 이유는 다음과 같다.

### 2.1 조회 빈도가 높고 중복 요청 가능성이 높은 기능

콘서트 예매 시스템의 특성상 사용자는 다음과 같은 행동을 반복적으로 수행한다.

* 특정 콘서트의 날짜 목록을 여러 번 조회
* 동일한 콘서트 날짜에 대해 좌석 상태를 반복적으로 확인

이 과정에서 **동일한 파라미터의 요청이 짧은 시간 내 다수 발생**할 가능성이 높으며,
이는 캐시 도입 효과를 검증하기에 가장 적합한 조건이다.

---

### 2.2 읽기 위주의(Read-heavy) UseCase로 캐시 효과가 명확함

선정한 두 UseCase는 공통적으로 다음과 같은 특성을 가진다.

* 데이터 변경이 자주 발생하지 않음
* 읽기(조회) 요청 비중이 매우 높음
* 동일한 요청 결과를 재사용할 수 있음

이러한 특성은 캐시 적용 전·후의 성능 차이를 명확히 비교할 수 있게 하며,
캐시 도입 효과가 수치적으로 드러나기 쉬운 영역이다.

---

### 2.3 DB 부하 및 병목 가능성이 존재하는 핵심 지점

특히 좌석 조회 기능의 경우:

* 데이터베이스 조회 빈도가 높고
* 동시 요청이 몰릴 경우 DB 부하 증가 가능성이 큼
* 실 서비스 환경에서 성능 병목 지점이 될 가능성이 높은 영역이다

따라서 해당 기능을 테스트 대상으로 선정함으로써,
**캐시 적용이 응답 시간 개선뿐 아니라 DB 접근 횟수 감소에 어떤 영향을 미치는지** 확인하고자 했다.

---

### 2.4 시스템 전반의 체감 성능에 영향을 미치는 영역

콘서트 날짜 조회 및 좌석 조회는 사용자가 서비스에 진입했을 때
가장 먼저 호출되는 기능이며, 이후 예매·결제 흐름으로 이어지는 **전제 조건 역할**을 한다.

따라서 이 영역의 성능 개선은
단일 기능 최적화를 넘어 **시스템 전반의 체감 성능 개선으로 직결된다**고 판단하였다.

---

## 3. 캐시 적용 전략

* 캐시 저장소: **Redis**
* 적용 방식: Spring Cache 추상화 사용
* 캐시 대상:

    * 콘서트 날짜 조회 결과
    * 콘서트 좌석 조회 결과
* 캐시 키:

    * 콘서트 ID 또는 콘서트 날짜 ID 기반으로 설정
* 캐시 적용 위치:

    * 조회 전용 UseCase 레벨에 `@Cacheable` 적용

---

## 4. 성능 측정 방법

* 테스트 방식: SpringBoot 통합 테스트
* 동일한 요청을 **100회 반복 호출**
* 캐시 적용 전 / 캐시 적용 후 각각 측정
* 측정 지표: 총 소요 시간(ms)

---

## 5. 성능 측정 결과

### 5.1 콘서트 날짜 조회

| 구분      | 총 100회 조회 소요 시간 |
| ------- | --------------- |
| 캐시 적용 전 | 717ms           |
| 캐시 적용 후 | 189ms           |

➡ 약 **73% 이상 응답 시간 감소**

---

### 5.2 콘서트 좌석 조회

| 구분      | 총 100회 조회 소요 시간 |
| ------- | --------------- |
| 캐시 적용 전 | 761ms           |
| 캐시 적용 후 | 168ms           |

➡ 약 **78% 이상 응답 시간 감소**

---

## 6. 결과 분석

캐시 적용 이후 두 시나리오 모두에서 **유의미한 성능 개선**이 확인되었다.

* 반복 조회 시 DB 접근이 제거되면서 응답 시간이 크게 감소
* 조회 빈도가 높은 구간일수록 캐시 효과가 더욱 명확하게 나타남
* 특히 좌석 조회의 경우, DB 부하 및 잠재적 병목 지점을 효과적으로 완화할 수 있음을 확인하였다

이는 Redis 기반 캐싱 전략이
**읽기 위주의 도메인에서 매우 효과적인 성능 최적화 수단임을 보여준다.**

---

## 7. 결론

본 과제에서는 조회 빈도가 높고 중복 요청 가능성이 큰 핵심 기능을 선별하여
Redis 캐싱 전략을 적용하고 성능 개선 효과를 검증하였다.

그 결과, 제한된 테스트 범위 내에서도
**응답 시간 단축 및 시스템 부하 감소라는 명확한 성능 개선 효과**를 확인할 수 있었으며,
실 서비스 환경에서도 충분히 적용 가치가 있는 전략임을 확인하였다.
